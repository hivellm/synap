# Project Rules

Generated by @hivellm/rulebook. Spec files are in `/.rulebook/specs/`.

---

## Critical Rules (Highest Precedence)

1. **Read `AGENTS.md` and `AGENTS.override.md`** before making changes.
2. **Never revert or discard uncommitted work** — fix forward. Treat the working tree as sacred.
3. **Edit files sequentially**, not in parallel. When a task touches 3+ files, decompose into 1–2 file sub-tasks.
4. **Run `check`/type-check before `test`** — diagnostics are 5–10x faster and catch errors tests miss.
5. **If a fix fails twice, escalate** — stop, research, or open a team. Never try the same approach a third time.
6. **Prefer MCP tools** (`mcp__rulebook__*`) over shell commands when an equivalent exists.
7. **Capture learnings** to `.rulebook/knowledge/` and `.rulebook/learnings/`.
8. **Never archive a task** without docs updated, tests written, and tests passing.

## Tier 1 — Absolute Prohibitions

These override ALL other rules. Violation = output rejected.

### 1. No Shortcuts, Stubs, or Simplified Logic
**NEVER** simplify logic, add TODO/FIXME/HACK, create stubs, use placeholders, alter existing logic to avoid complexity, reduce scope, skip edge cases, or deliver partial implementations. Research the correct approach, implement completely, ask if unsure.

### 2. No Destructive Git Operations Without Authorization
**Allowed:** `git status/diff/log/blame`, `git add/commit` (after quality checks).
**Forbidden without authorization:** `git stash/rebase/reset --hard/checkout -- ./restore ./revert/cherry-pick/merge/branch -D/push --force/clean -f/checkout <branch>/switch`. Multiple AI sessions may share the same worktree.

### 3. No Deletion Without Authorization
**NEVER** `rm`/`del` any file without explicit user authorization.

### 4. Research Before Implementing — Never Guess
State what you KNOW → state what you DON'T KNOW → research the unknown → only then implement. "I think this might be the problem" is NOT acceptable.

### 5. Sequential File Editing
Read file1 → Edit file1 → Read file2 → Edit file2. Never batch-read then batch-edit. When 3+ files: decompose into 1–2 file sub-tasks in dependency order.

### 6. No Deferred Tasks
Implement every checklist item. If a dependency exists, implement it FIRST. Exception: deferred items MUST have a new tracking task created BEFORE archiving.

### 7. Follow Task Sequence — No Reordering
The task list is an ORDER, not a MENU. Execute items top to bottom. Never skip, reorder, or start Phase N+1 before Phase N is 100% complete.

## Quality Enforcement (Non-Negotiable)

- **NEVER** bypass tests (`.skip()`, `.only()`, comment out, `@ts-ignore`, mock-to-pass)
- **NEVER** bypass git hooks (`--no-verify`, `-n`, `--no-gpg-sign`)
- **NEVER** write empty/boilerplate tests or tests without assertions
- **NEVER** create temp files outside `/scripts` — clean up immediately
- Fix root causes, not symptoms. **Violation = Implementation Rejected.**

## Task Workflow

**MANDATORY: Use Rulebook MCP tools** (`rulebook_task_create/update/archive/list/show/validate`).

1. `rulebook_task_list` → pick first unchecked item from lowest phase
2. Read `proposal.md` and `tasks.md` before coding
3. Implement step by step, run lint + type-check after each change
4. Mark items `[x]`, `rulebook_task_update` status
5. Mandatory tail (docs + tests + verify) — `rulebook_task_archive` enforces this

### Task Creation (MANDATORY before implementation)
1. `rulebook_task_create <task-id>` (verb-led kebab-case, e.g. `add-user-auth`)
2. Write `proposal.md` (why ≥20 chars, what, impact) and `tasks.md` (checklist only)
3. Write spec deltas in `specs/<module>/spec.md` (SHALL/MUST + Given/When/Then)
4. `rulebook_task_validate <task-id>`, then implement

### Task File Structure
```
.rulebook/tasks/<task-id>/
├── proposal.md    # Why and what
├── tasks.md       # Checklist ONLY (- [ ] / - [x])
├── design.md      # Optional
└── specs/<module>/spec.md  # SHALL/MUST requirements
```

Lifecycle: Create → Plan → Design → Spec → Validate → Implement → Test → Document → Archive.

## Persistent Memory & Knowledge Base

### Memory
Save via `rulebook_memory_save({ type, title, content, tags })`. Search via `rulebook_memory_search({ query, mode: "hybrid", limit: 10 })`. Session: search at start, save during, `rulebook_session_end` at end.

### Knowledge Base
**Before** implementing: `rulebook_knowledge_list`, `rulebook_learn_list`, `rulebook_decision_list`.
**After** implementing: capture ≥1 entry per task via `rulebook_knowledge_add`/`rulebook_learn_capture`/`rulebook_decision_create`.

### Decision Records (ADRs)
In `.rulebook/decisions/`. Lifecycle: proposed → accepted → superseded | deprecated. Never delete.

## Agent Automation Workflow

After completing ANY implementation:

1. **Quality checks**: type-check → lint (zero warnings) → format → tests → coverage (≥95%)
2. **Memory**: save learnings via `rulebook_memory_save`
3. **Security**: `cargo audit` — never ignore critical/high without user approval
4. **Tasks**: mark `[x]`, update status
5. **Docs**: update CHANGELOG.md, README if public API changed
6. **Commit**: `git add <files>` + conventional commit (`feat/fix/docs/refactor/perf/test/build/ci/chore`), English only
7. **Report**: `✅ Done. Committed: <hash>` — no emoji tables, no "Next Steps"

## Documentation Standards

All docs in English. Root-level: `README.md`, `CHANGELOG.md`, `AGENTS.md`, `LICENSE`, `CONTRIBUTING.md`, `SECURITY.md`. Everything else in `/docs`.

## Rust Project Rules

- **Edition**: 2024, **Toolchain**: nightly 1.85+
- **Quality sequence**: `cargo fmt` → `cargo clippy -- -D warnings` → `cargo test --all-features` → `cargo build --release`
- **Testing**: unit tests in `#[cfg(test)]`, integration in `/tests`, coverage ≥95%, async with `#[tokio::test]`
- **Test categories**: fast (default), `--features s2s` (needs server), `--features slow` (>20s)
- **Error handling**: `thiserror` for libraries, `anyhow` for apps, `Result<T,E>` always, no `unwrap()` in production
- **Async**: Tokio runtime, never block, `tokio::time::timeout` on network ops, `spawn_blocking` for CPU work
- **Dependencies**: verify on crates.io, `cargo audit`, minimize count
- **Docs**: `///` on public APIs with examples, `//!` on modules, `#![deny(missing_docs)]` for libs

## DAG Architecture

Foundation → Core → Features → Presentation. Lower layers MUST NOT depend on higher layers. No circular dependencies. Verify with `cargo check`/`cargo tree`.

## Multi-Agent Teams

Use Teams (`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`) for parallel multi-agent work — standalone background agents cannot communicate. Background `Agent` calls require either `subagent_type: team-lead` or `team_name` set.

Standard team: team-lead (orchestrator), researcher (haiku), implementer (sonnet), tester (sonnet). Use haiku for read-only tasks. Primary language: **Rust**.
