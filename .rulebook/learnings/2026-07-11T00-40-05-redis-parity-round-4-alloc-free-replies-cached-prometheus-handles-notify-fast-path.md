# Redis-parity round 4: alloc-free replies + cached prometheus handles + notify fast path
**Source**: manual
**Date**: 2026-07-11
phase12 round 4 patterns that moved -P16 throughput (INCR 746k->793k, LPOP 750k->815k, RPUSH 678k->753k): (1) RESP3 reply headers/integers via itoa-style back-to-front stack formatter (fmt_num_line, [u8;24], prefix+digits+CRLF) replacing format! — Redis's ll2string equivalent; one String alloc removed from EVERY reply. SimpleString/Error write in 3 parts — fine because the sink is a BufWriter (each write_all is a memcpy, not a syscall). (2) prometheus with_label_values is NOT cheap: label hash + internal RwLock read per call, and we called it 3x/command; fix = pre-resolve children into OnceLock<HashMap<&'static str, handles>> for the hot command set (children are cheap Arc clones), fallback to dynamic path for unknown commands. (3) notify_waiters (blocking-pop wakeup) did RwLock read + HashMap lookup on EVERY list/zset push even with zero waiters; fix = AtomicU64 channel counter, skip when 0 — dead cost for non-blocking workloads. Meta-lesson: at ~800k ops/s, ONE String alloc or ONE HashMap-behind-RwLock lookup per op is a measurable single-digit-% tax; hunt them with 'what does this op do that Redis does not'. Mission tasks for the remaining gap: phase13_parse-bulk-into-arc (SET double-copy), phase13_int-encoding-counters (INCR zero-alloc), phase13_contiguous-list-encoding (RPUSH listpack), phase13_multikey-benchmark-sweep (show sharding advantage).
