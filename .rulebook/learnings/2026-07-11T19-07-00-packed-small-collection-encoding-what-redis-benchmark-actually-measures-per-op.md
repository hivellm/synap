# Packed small-collection encoding + what redis-benchmark actually measures per op
**Source**: manual
**Date**: 2026-07-11
phase13 contiguous encoding, both halves. Pattern that contained the risk: dual representation (Packed{buf,count} with [u32 LE len][bytes] entries | original structure) where ONLY hot ops get native packed paths and every complex mutator calls ensure_deque()/ensure_hash() — a lazy ONE-WAY upgrade — then runs the existing tested logic unchanged. Serde stays format-compatible via a custom `with` module that serializes the repr as the logical element sequence (serialize_seq), identical to what VecDeque/HashSet derive produced — old snapshots load, new ones stay readable (validated by existing persistence round-trip tests). Public field -> private repr behind methods; external users (recovery into_members/into_elements, memory accounting member_bytes/element_bytes) get accessors. Result: LPUSH multi-key 1.22-1.46x Redis. CRITICAL benchmark literacy learned: (1) redis-benchmark -t sadd uses a FIXED key ('myset') with randomized members even under -r — it measures big-set insert (150k members into one set: HashSet resize pauses vs Redis incremental rehash), NOT small-set creation; small-collection encodings won't move it. Only set/get/incr use key:__rand_int__. (2) Long-lived benchmark server containers accumulate millions of keys across runs and silently skew comparisons — restart BOTH servers for any final numbers. (3) Redis listpack has a big append/prepend asymmetry (RPUSH 955k vs LPUSH 537k on the same run); when comparing directional list ops, compare against the matching direction and note the asymmetry, or a symmetric implementation (~650k both ways) looks falsely bad on one side.
