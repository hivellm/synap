# Static port counters in Rust integration tests break under nextest (per-process execution)
**Source**: manual
**Date**: 2026-07-12
**Tags**: rust, testing, nextest, flaky-tests, ci, anti-pattern, replication
Anti-pattern: `static PORT: AtomicU16 = AtomicU16::new(BASE); fn next_port() { PORT.fetch_add(1, ..) }` in tests/*.rs files that bind TCP listeners. It is collision-free under `cargo test` (tests are threads in one process) but CI runs `cargo nextest`, which executes EVERY test in its own process — each process restarts the counter at BASE, so all tests in the file race to bind the same port. Symptom: intermittent CI-only failures where exactly one test per wave passes and the rest hit poll/convergence deadlines (losers' replicas connect to the winner's master; their own writes go nowhere). Deterministic local repro: `cargo nextest run --test <suite>` (Synap datatype_replication_tests went 1/4 passing). Fix: ask the OS for an ephemeral port — `std::net::TcpListener::bind("127.0.0.1:0").unwrap().local_addr().unwrap().port()` — which is collision-free across processes. Fixed in 5 suites (datatype/kv/stream replication, replication_integration, transaction_durability), 39/39 green under nextest. General rule: any per-process resource uniqueness scheme (ports, temp dirs like ./target/fixed-name, shared counters) must assume nextest's process-per-test model.
