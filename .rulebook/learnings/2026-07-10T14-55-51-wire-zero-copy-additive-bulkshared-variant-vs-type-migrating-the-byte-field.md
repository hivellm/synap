# Wire zero-copy: additive BulkShared variant vs type-migrating the byte field
**Source**: manual
**Date**: 2026-07-10
phase11 wire-value zero-copy — two protocols, two different tactics based on churn/risk. RESP3 Resp3Value::BulkString(Vec<u8>) had 121 construction sites; instead of migrating them all to Arc<[u8]> (most build small owned reply values that would then pay a Vec->Arc copy anyway), I ADDED a Resp3Value::BulkShared(Arc<[u8]>) reply-only variant used solely by the GET/MGET hot path. Writer serializes it byte-for-byte like BulkString; as_bytes/as_str handle both. Zero churn on existing sites, and GET now carries the store's get_shared() Arc straight to the socket. Downside: BulkShared != BulkString under derived PartialEq, so GET/MGET dispatch tests had to switch from asserting the exact variant to `matches!(_, BulkShared(_))` + `.as_bytes()`. SynapRPC SynapValue::Bytes had only 37 sites, so I type-migrated it to Arc<[u8]> — but serde can't derive Serialize/Deserialize for Arc<[u8]> without the global `rc` feature, so I added a localized `#[serde(with = "arc_bytes")]` adapter that serializes (**bytes) as [u8] (identical wire to Vec<u8> under rmp-serde) and deserializes via Vec then Arc::from. Kept `impl From<Vec<u8>>` (now does .into()) + added `From<Arc<[u8]>>` so most sites become SynapValue::from(x) (works for both Vec and Arc); numeric-literal test sites like vec![0xff,0x00] need explicit u8 typing (vec![0xffu8,..].into()) since From<Vec<i32>> doesn't exist. mget stays a thin to_vec wrapper over a new mget_shared. The existing rmp_serde round-trip test (wire_roundtrip_all_variants) is the wire-compat proof.
