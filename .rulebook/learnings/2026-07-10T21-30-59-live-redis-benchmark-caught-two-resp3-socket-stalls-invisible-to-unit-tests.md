# Live redis-benchmark caught two RESP3 socket stalls invisible to unit tests
**Source**: manual
**Date**: 2026-07-10
Running the actual Redis head-to-head (redis-benchmark against Synap's RESP3 listener in Docker) exposed two socket-layer perf bugs that no unit/integration test caught because they only manifest as wall-clock throughput under a real TCP client: (1) missing TCP_NODELAY — a RESP3 bulk reply is written as 3 segments (len header, payload, CRLF); with Nagle on, the payload waits ~40ms for the header's delayed ACK, so non-pipelined GET/LRANGE ran at ~1085 rps while single-segment writes (SET) were fine at ~52k. Fix: stream.set_nodelay(true) after accept. (2) The Resp3Writer wrapped the raw OwnedWriteHalf, so the 'pipeline-aware flush' (flush only when reader.buffer() empty) was a no-op — flush() on a raw TcpStream does nothing and every segment was its own write(); pipelined -P16 was capped flat at ~17k rps for ALL ops. Fix: wrap the write half in tokio::io::BufWriter so the deferred flush coalesces a batch into one syscall. Combined result: GET -P1 1085->56k (parity with Redis), -P16 17k->833k (~90% of Redis). Lesson: for a network server, a live protocol benchmark is a REQUIRED test class — correctness tests pass while throughput is 50x off. Also found: pipelined single-hot-key SET/INCR bottlenecked at ~130k by the per-key async tokio::Mutex (audit M-010 EXEC isolation) vs 740k for parking_lot-guarded collection writes — async per-key locks are expensive under contention. Docker gotcha: the release Dockerfile's musl static build fails in this env; a glibc rust:1-bookworm builder + debian-slim runtime (scripts/Dockerfile.bench) builds cleanly. .dockerignore excludes scripts/ and config*.yml so the -f Dockerfile is read from host but COPY sources must live elsewhere (used config/bench-config.yml).
